# GComp - разбор и сборка .ert / 1Cv7.MD

GComp - утилита для декомпиляции/компиляции бинарных файлов 1С 7.7 (`1Cv7.MD`, `.ert`)
в текстовые исходники и обратно. Версия в проекте - 2.2.15.beta. (`.erf` - формат 8.x, к 7.7 не относится.)

## Рабочий GComp (критично)

Запускать **только полную установку** (exe + `GComp.dll` + `MFC42.DLL` рядом):

```
C:\Program Files\GComp\Release\gcomp.exe      ← основной, рабочий
```

Запасные комплекты (если основной недоступен):
- `C:\Tools\gcomp\`
- `C:\1Cv77\BIN\` (рядом с платформой)

**Не использовать «голый» gcomp.exe** из каталогов проекта (например
`Конфигурации\...\KonturEDI\АнализКонфигурации\gcomp.exe`) - рядом нет `GComp.dll`, и он падает с
`STATUS_DLL_NOT_FOUND` (`EXIT=-1073741515` / `0xC0000135`). Это компонент сторонних обработок, не наш.

В каталоге Release также есть `vgcomp.exe` - GUI-версия (визуальная), для скриптовой работы не нужна.

## Запускать только из PowerShell

Из Bash кириллица в путях (`ВыгрузкаИзТис77`, `PUBID_NNNN-...`) **бьётся**: gcomp получает битые
аргументы, ничего не находит и **молча выходит с EXIT 0** - создаётся ложное впечатление успеха, а файл
не меняется. Поэтому:

- Запуск - через PowerShell (нативная Windows-кодировка), либо через `scripts/build_ert.py` /
  `scripts/unpack_ert.py`, которые вызывают gcomp правильно.
- После сборки **обязательно** проверь, что у `.ert` изменился timestamp/размер. Не меняется - значит
  аргументы не дошли.

## Команды

GComp штатно не печатает usage (выдаёт `exit 53` без вывода); справка ниже извлечена из бинарника.

### Разбор (декомпиляция)

```
gcomp -d -F <файл.ert> -DD <каталог_назначения>
```
- `-d` - режим разбора.
- `-F <файл>` - что разбирать (`.ert`, или опустить `-F` для `1cv7.md` в текущем каталоге).
- `-DD <каталог>` - куда положить исходники. Без `-DD` имя каталога формируется как `SRC\<ИмяБезРасширения>`.
- Предупреждение GComp: разбор в непустой каталог может затереть данные - разбирай в чистый каталог.

### Сборка (компиляция)

```
gcomp -c -F <файл.ert> -DD <каталог_исходников>
```
- `-c` - режим сборки.
- `-F <файл>` - целевой `.ert` (перезаписывается молча - делай бэкап заранее).
- `-DD <каталог>` - откуда брать исходники. Без `-DD` берётся `SRC\<ИмяБезРасширения>`.

### Полезные опции

- `--external-report` - явно указать, что работаем с внешним отчётом/обработкой (`.ert`). Обычно
  определяется по расширению `-F`, но можно задать явно.
- `--meta-data` - работаем с файлом конфигурации (`1cv7.md`).
- `--truncate-mms` - при сборке `.ert` заменить поток `Main MetaData Stream` на пустой (уменьшает размер,
  если копия метаданных конфигурации в обработке не нужна). По умолчанию НЕ труначить, если обработка
  опирается на типы реквизитов из этой структуры.
- `--no-parse-dialogs` - выгрузить формы как есть, без обработки (для проблемных диалогов).
- `--more-subfolders` - разбивать на каталоги объекты, которые по умолчанию идут одним `.mdp`.
- `-v` / `-vv` - подробный вывод (для диагностики сборки).

## Безопасный порядок работы

1. **Бэкап** целевого `.ert` (gcomp перезаписывает молча):
   `Copy-Item <файл.ert> <файл.ert>.bak_<дата>`
2. Сборка из исправленного каталога исходников (`build_ert.py` или PowerShell напрямую).
3. Проверка: timestamp и размер `.ert` изменились; при сомнении - разобрать собранный `.ert` во временный
   каталог и сверить, что правка на месте (round-trip).
4. Финальная компиляция BSL и проверка - при открытии обработки в 1С 7.7 (gcomp пакует, но не проверяет
   синтаксис BSL так, как платформа).

## Пример (PowerShell)

```powershell
$g   = "C:\Program Files\GComp\Release\gcomp.exe"
$out = "E:\...\src\PUBID_NNNN_ert\ВыгрузкаИзТис77.ert"
$src = "E:\...\src\PUBID_NNNN_ert\PUBID_NNNN-ВыгрузкаИзТис77"
Copy-Item $out "$out.bak" -Force
& $g -c -F $out -DD $src
"EXIT=$LASTEXITCODE"
Get-Item $out | Select-Object Name,Length,LastWriteTime
```

Целевое имя выходного `.ert` для проекта миграции - `ВыгрузкаИзТис77.ert` (без префикса `PUBID_NNNN-`);
каталоги исходников `PUBID_NNNN_ert/` не переименовываются - это маркер источника.
