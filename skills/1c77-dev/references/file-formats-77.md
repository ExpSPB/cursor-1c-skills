Что внутри бинарных `.md`/`.ert` и как устроены текстовые исходники после разбора GComp.

## 1. Бинарные контейнеры

- **`1Cv7.MD`** - вся конфигурация (метаданные + код + формы + макеты + права) в одном бинарном файле
  формата Compound Document (как старый .doc). Внутри - потоки в скобочном синтаксисе, CP1251.
- **`.ert`** - внешняя обработка/отчёт. В 7.7 это единый формат (в отличие от 8.x, где `.epf`/`.erf`).
  Тот же Compound Document, несёт форму, модуль, макет и (часто) копию метаданных конфигурации.

Напрямую эти файлы не редактируются. Их **разбирают** GComp в дерево текстовых файлов, правят, **собирают**
обратно. См. `references/gcomp.md`. Альтернатива для чтения метаданных без разбора - MCP `1c77-metadata`,
который парсит `1Cv7.MD` и отдаёт объекты/модули через API (см. `references/mcp-and-tools.md`).

## 2. Состав разобранной обработки .ert

После `unpack_ert.py` (GComp `-d`) каталог обработки содержит примерно:

```
<Обработка>/
├── МодульФормы.1s          # код обработки (CP1251) - основная цель правок
├── Форма.frm               # диалог формы (CP1251)
├── Структура.mdp           # метаданные: реквизиты обработки + нередко копия всей конфигурации
├── Таблица.mxl             # макет табличного документа (печать/вывод)
├── ОбъектыМетаданных.txt    # реестр объектов (Справочник.X, Документ.Y ...) с числовыми ID
├── gcomp.ini               # параметры разбора (язык, флаги)
├── gcomp.ver               # версия GComp, которой разбирали
└── ПорядокОбъектов.ord      # порядок объектов для детерминированной сборки
```

Размер `Структура.mdp` может быть сопоставим с размером всего `.ert` - это значит обработка несёт
полную метаструктуру конфигурации (нужна для разбора путей реквизитов вроде `спр.Товар.Артикул`).
При сборке такую структуру сохраняют как есть (без `--truncate-mms`), иначе обработка может потерять
сведения о типах. Если структура не нужна - `--truncate-mms` уменьшает результат.

## 3. Модули .1s

Обычный текст BSL 7.7 в CP1251. Имена в типовых распаковках:

- `ГлобальныйМодуль.1s` - глобальный модуль конфигурации (общие функции, в т.ч. `ПустаяСтрока` и пр.).
- `Документы/<Имя>/МодульПроведения.1s`, `.../МодульФормы.1s`.
- `Справочники/<Имя>/МодульФормы.1s`.
- `Обработки/<Имя>/МодульФормы.1s`, для внешней обработки - просто `МодульФормы.1s`.

Правка - только через CP1251 (`scripts/cp1251_io.py`). Обычный Grep по кириллице в `.1s` ничего не находит
(он ищет UTF-8-байты). Синтаксис - см. `references/syntax-gotchas-77.md`.

## 4. Формы .frm

Скобочный текстовый формат (CP1251). Описывает диалог: параметры окна и список элементов управления.

```
Диалог:
{
    Параметры формы:
    {
        Параметры:
        {
            Ширина: 322
            Высота: 263
            Заголовок: " "
        }
    }
    Элементы:
    {
        BUTTON:
        {
            Заголовок: "1. Выгрузить склады"
            ИдентификаторМетаданных: -1
            ДопФлаги: Flag0, Flag1, КартинкаПоЦентру, Flag28, Flag30
            X: 7
            Y: 38
            Ширина: 148
            Высота: 14
            ПорядокОбхода: 4200
            Формула: "Демо_ВыгрузитьСклады()"   // обработчик: вызов процедуры модуля
            Слой: Основной
        }
        // другие элементы: TEXT, EDIT, CHECKBOX, GROUP, TABLE и т.д.
    }
}
```

Ключевое для правок:
- **Кнопка вызывает процедуру** через поле `Формула: "ИмяПроцедуры()"`. Добавить кнопку = вписать блок
  `BUTTON` с координатами `X/Y/Ширина/Высота`, уникальным `ПорядокОбхода` и `Формула`, плюс саму процедуру
  в `.1s`.
- Координаты в диалоговых единицах; следи, чтобы новые элементы не накладывались (проверяй визуально после
  открытия в 7.7).
- Правка через CP1251. Для добавления повторяющихся элементов удобен `patch_by_marker.py`.

## 5. Метаданные .mdp

Скобочный текстовый формат (CP1251). Описание объектов конфигурации/обработки: реквизиты, табличные части,
свойства. Шапка несёт служебные счётчики (`Next ID`, `Версия`).

```
Main Metadata Stream:
{
    MainDataContDef: { Next ID: 9144  Версия: 10009 ... }
    Описание задачи: { { Идентификатор: "..."  Комментарий: "7.70.927" ... } }
    Справочники:
    {
        Банки:
        {
            Синоним: "Банки (классификатор БИК)"
            Длина кода: ...
            Длина наименования: ...
            Количество уровней: 1
            // реквизиты, табличные части ...
        }
    }
    Документы: { ... }
    Регистры: { ... }
}
```

Ручная правка `.mdp` рискованна (числовые ID, счётчики, ссылки между объектами). Для **анализа** структуры
объекта лучше MCP `1c77-metadata` (`get_object` - реквизиты и табличные части без ручного парсинга).
Менять метаданные надёжнее в самой 7.7 (Конфигуратор), а не правкой `.mdp` руками. Если правишь `.mdp` -
не нарушай скобочную структуру и отступы, иначе сборка GComp упадёт или даст битый объект.

## 6. Макеты .mxl

Табличный документ (печатные формы, вывод обработок). После разбора - текстовый, но громоздкий. Для
точечной работы с печатью обычно достаточно знать имена областей и параметров. Глубокая правка `.mxl`
руками неудобна - чаще проще отредактировать макет в 7.7.

## 7. Прочие служебные файлы

- `gcomp.ini` - флаги разбора (язык Rus, `no-parse-forms`, `more-subfolders` и т.п.). При сборке влияет
  на трактовку форм/папок.
- `gcomp.ver` - версия GComp. Полезно собирать той же версией, что разбирали.
- `*.ord` - порядок объектов для детерминированной сборки.
- `ОбъектыМетаданных.txt` - человекочитаемый реестр объектов с ID; удобен для быстрой проверки, какие
  объекты есть и как называются.
